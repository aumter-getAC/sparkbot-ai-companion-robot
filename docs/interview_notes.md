# 保研面试讲法与问答

## 1. 项目介绍模板

老师您好，这个项目是我基于 ESP-SparkBot / sparkbot-45coll 平台复现并改造的桌面陪伴机器人。原平台提供了 ESP32-S3 头部主控、屏幕、音频、摄像头、触摸和小车底盘等基础能力。我主要做的是大模型应用层改造：将机器人原有 AI 对话链路接入 Qwen API，并通过提示词工程、轻量知识库检索和动作意图解析，让机器人能够自动回答用户问题、提供情感支持，并根据用户需求触发小车运动或服务功能。这个项目的价值在于把通用大模型能力落到一个资源受限的嵌入式机器人上，形成了“输入感知—模型理解—知识增强—语音/屏幕输出—动作执行”的完整闭环。

## 2. 你完成了什么

可以说：

- 我完成了 Qwen API 的接入和 OpenAI-compatible 接口适配；
- 我设计了桌面机器人角色 prompt，使回复短、自然、适合语音播报；
- 我加入了轻量知识库，让机器人能回答和自身功能、用户需求有关的问题；
- 我做了自然语言到控制命令的映射，例如前进、停止、巡线、拍照；
- 我处理了联网超时、API Key 管理、模型回复过长、TTS 播放不自然等问题。

不要说：

- “我训练了 Qwen 模型”；除非你真的做过微调。
- “我完全独立开发了整个机器人硬件”；应说明基于开源项目复现并二次开发。

## 3. 可能被问的问题

### Q1：你为什么不用 ESP32 直接跑大模型？

ESP32-S3 算力和内存不足以运行真正的大语言模型，所以我的设计是边缘设备负责语音、显示、传感器和动作控制，把大模型推理放到云端 Qwen API 或自建 AI Bridge。这样既保留机器人实时交互，又避免在微控制器上承担不现实的推理负载。

### Q2：Qwen API 接入改了哪些地方？

主要改三类：第一是接口参数，包括 base_url、api_key、model；第二是请求体，从用户输入、历史对话和 system prompt 组合成 messages；第三是响应解析，把 assistant 的回复提取出来，同时处理超时、错误码、回复长度，并把结果传给屏幕和 TTS。

### Q3：知识库是怎么实现的？

复现阶段采用轻量文本检索：把项目说明、用户偏好和常见问答整理成 Markdown，用户提问时做关键词匹配，选出相关片段放进 system prompt。升级方案可以用 embedding 模型把文档向量化，存入向量数据库，再做相似度检索，这就是标准 RAG 思路。

### Q4：如何让它具备情感支持能力？

不是靠硬编码安慰话术，而是在 system prompt 中规定机器人角色、语气和边界：先识别用户情绪，再共情，最后给一个可执行的小建议。同时控制回答长度，让它像桌面机器人说话，而不是长篇大论。

### Q5：怎么控制小车？

用户说“前进、后退、停止、巡线”等话语后，服务端或固件把自然语言识别为结构化 command，例如 `{type: move, action: forward}`。ESP32-S3 收到后再通过原项目的小车通信链路转发给 ESP32-C3 底盘控制模块。

### Q6：项目难点是什么？

难点主要有四个：一是 ESP32 资源有限，不能把上下文和知识库都塞进设备；二是云 API 调用有延迟，需要优化超时和回复长度；三是大模型回复不可控，需要通过 prompt 和命令解析限制它的行为；四是硬件功能多，语音、屏幕、Wi-Fi、小车和摄像头之间要协调状态。

### Q7：这个项目的创新点是什么？

不是硬件本身原创，而是把开源机器人平台改造成具备个人化服务能力的智能体。创新点在于：Qwen 大模型接入、知识库增强、桌面陪伴型 prompt、自然语言控制小车，以及把 AI 对话和实体动作执行形成闭环。
