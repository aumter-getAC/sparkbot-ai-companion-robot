# 工程代码核心工作梳理

## 1. 原项目核心

SparkBot / sparkbot-45coll 的本质是一个边缘智能机器人平台：

- 头部 ESP32-S3：负责 Wi-Fi、屏幕、摄像头、音频、触摸、IMU、AI 对话页面；
- 小车 ESP32-C3：负责底盘运动、巡线、网页控制等；
- 软件层：天气、2048、木鱼、骰子、小车网页控制、小车巡线、AI 对话、AI 对话控制小车、摄像头等功能；
- 硬件层：在原 ESP-SparkBot 基础上优化供电，加入无线充电、小车巡线、一键开关、触摸按键扩展等。

## 2. 你复现/改造的核心

你的工作不应表述为“我训练了一个大模型”，而应表述为：

> 我在开源桌面机器人平台上做了大模型应用层集成与机器人交互适配：把 Qwen API 接入原有 AI 对话链路，并通过提示词工程、知识库检索、上下文管理和动作意图解析，把通用大模型改造成适合桌面陪伴机器人的服务型助手。

核心任务拆解：

1. **模型接口替换/适配**
   - 将原先 OpenAI/ChatGPT 风格请求改为 Qwen / DashScope；
   - 配置 base_url、api_key、model；
   - 调整请求 JSON、响应解析、超时与错误处理。

2. **机器人角色调校**
   - 设计 system prompt：身份、语气、能力边界、情绪支持方式、回答长度；
   - 让回复适合 TTS 播放：短句、口语化、少 Markdown；
   - 避免机器人说出自己不能执行的能力。

3. **知识库能力**
   - 把项目说明、用户偏好、服务规则、常见问答整理为 Markdown/文本；
   - 用户提问时检索相关片段，拼接进 prompt；
   - 让机器人回答带有“自己的知识”，而不是只依赖通用模型。

4. **机器人服务能力**
   - 根据用户意图提供服务：学习建议、情绪安慰、日常提醒、项目讲解；
   - 对“前进、后退、左转、巡线、拍照”等意图转成结构化命令；
   - 命令再由固件发送给小车底盘或摄像头模块。

5. **工程化处理**
   - API Key 不写死在公开仓库；
   - 加入重试、超时、错误提示；
   - 保存对话历史但控制长度；
   - 服务端日志用于调试，但注意隐私脱敏。

## 3. 一句话技术路线

语音/触摸输入 → ESP32-S3 联网 → AI Bridge / Qwen API → 知识库增强与人格化提示词 → 回复文本 + 机器人动作命令 → 屏幕/TTS 输出 + 小车执行。
